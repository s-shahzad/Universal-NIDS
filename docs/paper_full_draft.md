# Hybrid Intrusion Detection Workspace: Full Paper Draft

## 1. Abstract

This paper presents a hybrid intrusion detection workspace that combines live network capture, offline PCAP replay, optional Suricata and Zeek ingest, artifact static analysis, deterministic signature detection, statistical anomaly detection, supervised machine learning, optional unsupervised anomaly scoring, and a fusion engine for alert adjudication. Rather than positioning the system only as a detection stack, the work emphasizes an evidence-driven validation model in which software testing, repeatable offline scenarios, prepared-environment live validation, benign adjudication, suppression checks, and operational readiness records are part of the contribution. In the current repository state, the project records 157 collected tests, a default result of 149 passing tests with 8 deselected, and 79.16% coverage under a CI-enforced floor of 72%. Validation currently covers five latest passing offline lab scenarios and thirteen prepared-environment scenarios in the latest view, of which twelve are passing and one is partial. The completed six-hour PREP-ENV-007 soak remained operational but ended partial with 37598 flows, 19 alerts, peak RSS 543268 KiB, peak CPU 161%, recorded storage growth 1343554193 bytes, and restart latency 13.329 seconds. The system is therefore best described as a controlled pre-deployment candidate rather than a production-ready deployment.

## 2. Introduction

Network intrusion detection remains difficult because practical environments require both detection breadth and operational credibility. Signature-only systems offer explainability but can miss novel or weakly expressed behaviors. Pure anomaly systems may detect unusual traffic but often impose false-positive burdens that undermine analyst trust. Machine-learning-heavy approaches can improve discrimination on labeled data, yet they still require evidence that they behave acceptably under realistic traffic, benign drift, and operator maintenance actions.

This project addresses that gap by treating a hybrid intrusion detection workspace and its validation process as a single research object. The repository is not only a collection of parsing, detection, and storage components; it is also a structured experimentation environment for evaluating how those components behave under offline replay, live capture, benign traffic, operator workflow changes, and sustained soak conditions. The goal is not to claim production readiness prematurely, but to show what an evidence-backed progression toward deployment confidence looks like in a research and professor-review context.

## 3. Related Work

Related work in network intrusion detection usually falls into several broad groups. The first includes signature-oriented IDS and rule engines, which are valuable for precision and analyst interpretability but limited against previously unseen patterns. The second includes statistical and anomaly-based systems, which improve sensitivity to unusual traffic but may struggle with benign distribution shifts. The third includes supervised machine learning, where classification quality depends heavily on feature design, label quality, and deployment realism. A fourth group combines multiple engines or data sources into hybrid systems intended to balance precision, recall, and interpretability.

The present workspace fits most closely into this hybrid category. Its distinction is not only the combination of signature detection, anomaly scoring, supervised ML, optional unsupervised ML, and fusion. The main differentiator is the validation framing: prepared-environment scenarios, benign adjudication, suppression validation, queue-loss accounting, restart-based operator workflows, and readiness documents are treated as primary artifacts. This orientation aligns the work more closely with deployment-minded research than with benchmark-only model reporting.

## 4. System Architecture

The workspace accepts traffic and artifacts from several sources. Live NIC capture allows direct observation of network activity in a prepared environment. Offline PCAP replay supports repeatable experimentation without live host dependency. Optional Suricata and Zeek ingest broaden compatibility with external sensor pipelines. Artifact or file intake extends the workspace beyond packet-only analysis and supports correlation between network evidence and suspicious files.

After ingest, packets and events pass through parser and normalization logic that shapes heterogeneous data into a consistent internal representation. Feature extraction then derives flow-level, statistical, and machine-learning-ready inputs. Detection engines operate on those normalized features through multiple paths: signature detection for deterministic rule hits, statistical anomaly detection for thresholded behavior, supervised ML for labeled-pattern inference, optional unsupervised ML for outlier-like activity, and a fusion engine that adjudicates component outputs. Results are persisted in SQLite and JSONL and exposed through dashboards, reports, and evidence bundles.

## 5. Detection Pipeline

The detection pipeline is intentionally layered. Signature detection provides clear, reviewable alerts where known behaviors or rules apply. Statistical anomaly detection captures rate and aggregation changes that are not necessarily rule-bound. Supervised ML contributes pattern discrimination informed by labeled examples. Optional unsupervised ML acts as a bounded auxiliary detector for cases where unusual behavior may not match known signatures or supervised classes. Fusion then aggregates the component views and determines whether the combined evidence should remain benign, informational, or alert-worthy.

This structure supports a pragmatic tradeoff. No single engine is treated as universally authoritative. Instead, the system retains component-level evidence so that analysts and reviewers can understand why an alert occurred, why an alert did not occur, or why a benign sample was still problematic. That design choice is especially important in the current repository state, where the benign false-positive story is part of the honest technical conclusion.

## 6. Validation Framework

The validation framework spans both code-level and environment-level evidence. At the software test layer, the repository currently records 157 collected tests with 149 passing by default and 79.16% coverage under a 72% CI-enforced floor. These tests exercise runtime, parsing, feature extraction, detection, storage, reporting, artifact handling, visualization, and utility scripts. The result is a codebase that is materially broader in tested surface area than a prototype limited to model notebooks or offline accuracy claims.

Beyond software tests, the repository includes five latest passing offline lab scenarios that cover scan activity, brute-force behavior, flood or burst traffic, mixed benign and malicious traffic, and artifact-plus-network correlation. Prepared-environment validation adds thirteen current scenarios spanning live capture, malformed-packet handling, restart recovery, queue-loss accounting, benign adjudication, operator workflow validation, suppression validation, and soak behavior. In the latest prepared-environment view, twelve scenarios pass and one remains partial.

This validation framing is central to the paper. The project does not treat deployment readiness as a binary claim derived from unit tests or offline metrics alone. Instead, it records both strengths and unresolved constraints, including long-duration alert volume, maintenance blind windows, and resource-behavior questions that remain open after the first completed soak.

## 7. Experimental Results

The current experimental record supports several conclusions. First, the repository demonstrates stable software-test behavior with 157 collected tests, 149 passing tests in the default suite, no active pytest warnings, and 79.16% coverage. Second, the latest offline scenario set provides repeatable passing evidence across five distinct attack and artifact correlation families. Third, prepared-environment evidence shows that the live-capture path, queue-loss accounting, restart recovery, suppression behavior, and operator workflows all function in the intended prepared environment.

Benign adjudication is more nuanced. Earlier evidence showed a residual unsupervised alert on the burstier browsing and collaboration sample represented by PREP-ENV-012. After the Phase 9 candidate changes, both PREP-ENV-011 and PREP-ENV-012 reran cleanly, with 1622 flows and 0 alerts for the SaaS polling mix and 1814 flows and 0 alerts for the browsing and collaboration mix. That improves the short-run benign picture materially, but it does not yet justify a universal claim that the unsupervised false-positive issue is closed.

The long-duration soak question is now better bounded but not closed. The completed six-hour PREP-ENV-007 soak remained operational for 21600 seconds and preserved restart continuity, but it ended partial with 19 alerts, including repeated DoS and unsupervised events that were too noisy for promotion. Phase 9 targeted reruns then showed two important improvements: the focused live DoS burst stayed at one operator-visible DoS alert across 2410 flows, and the earlier PREP-ENV-012 benign false positive did not reproduce. Those results justify a fresh full-duration soak rerun, but not a stronger readiness claim yet.

## 8. Limitations

The most important limitation is that the only completed full-duration soak on record still ended partial. As long as PREP-ENV-007 remains bounded by a 19-alert soak result rather than a clean rerun on the updated profile, the project cannot claim the strongest form of release confidence for a controlled deployment candidate. A second limitation is that the short-run benign improvements still need confirmation under another long-duration soak. A third limitation is maintenance behavior: restart-based workflows are acceptable for the intended controlled scope, but they still imply an approximately 13-second blind window and therefore do not justify any uninterrupted-coverage claim.

The work is also bounded by environment scope. The prepared-environment topology has produced meaningful evidence, but not every operating system, interface model, or field deployment condition has been validated. For that reason, the correct current verdict remains a controlled pre-deployment candidate rather than a production-ready system.

## 9. Future Work

Future work should begin by rerunning the long-duration soak on the updated Phase 9 candidate and then revisiting the promotion gate on the same frozen profile. The next question is whether the Phase 9 short-run improvements hold under sustained benign traffic, especially for repeated unsupervised ICMP-like episodes and long-run DoS burst clustering. Beyond that, the most important engineering direction is hot reload or an equivalent zero-downtime maintenance strategy for environments that cannot tolerate restart-based blind windows.

Research-facing future work also includes stronger publication packaging. The current repository is already strong enough for professor review, paper drafting, and research discussion. The next step is to convert that evidence into a fully polished set of figures, tables, captions, and redacted public artifacts while preserving the current conservative claims.

## 10. Conclusion

This project demonstrates a hybrid intrusion detection workspace that is best understood as a combination of system architecture and validation methodology. Its contribution lies not only in combining signatures, anomaly detection, supervised ML, optional unsupervised ML, and fusion, but also in showing how offline scenarios, prepared-environment validation, benign adjudication, suppression checks, and maintenance evidence can be structured into a defensible readiness record. In its current state, the workspace supports a controlled pre-deployment candidate verdict and meaningful academic discussion. It does not yet support a production-readiness claim, and this draft intentionally preserves that distinction because the only completed long-duration soak on record remains partial and still requires a rerun on the updated Phase 9 candidate.
